# MOHProject — Project Rules

## Product

Building the **MOH SHIELD Addition-of-Rider(s)** enhancement (Singapore health insurance) plus sections 2–6 (Medisave switch, Premium tables, Change of Plan, Change of Citizenship, FR flow).

**Specs are the source of truth** — see [`docs/specs/`](docs/specs/). Before touching a domain field, an enum, or a state transition, find the FR ID in the relevant section spec and cite it in the PR description.

- Reading order for domain work: [glossary](docs/specs/glossary.md) → [domain-model](docs/specs/domain-model.md) → the phase you're on ([`docs/specs/phases/`](docs/specs/phases/)).
- Every FR has a stable ID (`FR-AOR-030`, `BUG-2610000310P`, `PH-02`, …). **Never renumber.** If a requirement changes, edit the block in place and add a change-log row.
- New/updated requirement doc → use the `update-spec` skill, don't hand-merge.

## Tech stack

- **ASP.NET Core MVC** on .NET 8 (project: `MOHProject.Web`)
- **Entity Framework Core 8** with **SQL Server** provider
- **Code First + Migrations** — no Database First, no EDMX
- **No authentication** by default. Ask before adding Identity/JWT/cookies.
- **Hangfire** proposed for reminder scheduling (Phase 5) — SQL Server storage.

## Layout

```
MOHProject/
├── MOHProject.sln
├── MOHProject.Web/
│   ├── Controllers/     # <Entity>Controller.cs, thin — delegate to services if logic grows
│   ├── Data/
│   │   └── AppDbContext.cs   # single DbContext; register all DbSet<T> here
│   ├── Models/          # POCO entities + view models
│   ├── Migrations/      # generated by `dotnet ef` — do NOT edit by hand
│   ├── Views/<Entity>/  # Razor views
│   ├── wwwroot/         # static assets
│   ├── Program.cs       # DI + middleware pipeline
│   └── appsettings.json # ConnectionStrings:DefaultConnection
└── MOHProject.Tests/
    ├── Unit/            # xUnit, no I/O, no DB — must run under 1s each
    └── Integration/     # xUnit + Testcontainers.MsSql, real SQL Server per run
```

## Conventions

- Namespaces: `MOHProject.Web.<Folder>` (e.g. `MOHProject.Web.Models`, `MOHProject.Web.Data`).
- Entity primary key: `int Id { get; set; }`.
- Money/decimal columns: `HasColumnType("decimal(18,2)")` in `OnModelCreating`.
- Timestamps: `DateTime CreatedAt { get; set; } = DateTime.UtcNow;` — always UTC.
- Use nullable reference types (`string?` for optional).
- Validation via `System.ComponentModel.DataAnnotations` on the entity/view model.
- Views: strongly-typed with `@model`, no `ViewBag` for domain data.
- Async by default for I/O: `ToListAsync`, `FirstOrDefaultAsync`, `SaveChangesAsync`.

## Database

- SQL Server runs in Docker locally. Container name: `mohsql`, port `1433`.
- Apple Silicon → `mcr.microsoft.com/azure-sql-edge`. Intel → `mcr.microsoft.com/mssql/server:2022-latest`.
- Connection string lives in `appsettings.json` under `ConnectionStrings:DefaultConnection`.
- **Never commit real passwords.** The default `Your_password123` is a placeholder — before pushing, move real secrets to User Secrets (`dotnet user-secrets`) or env vars.

## Migration protocol

Any change to an entity or `AppDbContext` MUST be followed by a migration in the same change:

```bash
dotnet ef migrations add <DescriptiveName>   # e.g. AddProductStockColumn
dotnet ef database update
```

Rules:
- Migration names are PascalCase and describe intent (`AddOrderTable`, not `Update1`).
- Never edit an already-applied migration — add a new one that adjusts.
- If a migration is wrong before being applied, delete it with `dotnet ef migrations remove` and regenerate.
- Do not run `EnsureCreated()` — migrations are the source of truth.

## Testing

- Framework: **xUnit** + **FluentAssertions 6.12** (last Apache-licensed version — do NOT upgrade to 7.x/8.x without confirming license impact) + **Moq**.
- Integration tests use **Testcontainers.MsSql** — a real SQL Server (Azure SQL Edge on Apple Silicon) spun up per test run. This matches the "no InMemory provider" rule.
- Docker must be running for `Integration/` tests. `Unit/` tests must not require Docker.
- Test class naming: `<ClassUnderTest>Tests`. Method naming: `Method_State_ExpectedResult`.
- Global usings: `Xunit` and `FluentAssertions` — do not re-import them per file.
- Every integration test picks its own unique data (`Guid.NewGuid()` in the name/key) — the fixture is shared across the collection and does NOT reset between tests. If you need reset semantics, wrap the test in a transaction and don't commit, or open a Respawn discussion before adding a global cleanup.
- Assertions include a **because** message whenever the assertion encodes an assumption about SQL Server behavior (collation, decimal precision, `DateTime.Kind`) — the message must name the assumption so a future reader knows what breaks.

## Common commands

```bash
# From MOHProject.Web/
dotnet build
dotnet run                    # HTTPS URL printed to console
dotnet ef migrations add <Name>
dotnet ef database update
dotnet ef migrations remove   # only if not applied

# From repo root
dotnet test                                            # all tests
dotnet test --filter "FullyQualifiedName~Unit"         # unit only (no Docker required)
dotnet test --filter "FullyQualifiedName~Integration"  # integration only (Docker required)
```

Ensure `dotnet-ef` is on PATH:

```bash
export PATH="$PATH:$HOME/.dotnet/tools"
```

## What NOT to do

- Do not scaffold controllers/views that overwrite hand-written files without asking.
- Do not add `Microsoft.EntityFrameworkCore.InMemory` — tests should hit a real SQL Server (Docker) to catch provider-specific issues.
- Do not introduce a second DbContext without a clear reason.
- Do not add packages targeting `net48` or older frameworks.
- Do not commit `appsettings.json` changes that swap in a real password — use secrets.
- Do not upgrade FluentAssertions past 6.12.x without checking the license — 7.x/8.x are commercial.
- Do not put I/O or DB access in `Unit/` — those belong in `Integration/`. A `Unit` test that touches disk/network is a bug in the test.
- Do not renumber a spec FR ID or reuse a retired ID. Deprecate in place instead — see [docs/specs/README.md](docs/specs/README.md).
- Do not merge a new requirement doc by hand-editing sections wholesale. Diff by FR ID; use the `update-spec` skill.
- Do not implement Midterm AOR (§1.7 · [FR-MID](docs/specs/sections/section-1-7-midterm.md)) by extending the NB/RN evaluator — Midterm is a separate bounded context with its own pipeline.
- Do not skip the 4 UAT bug regression tests ([BUG-uat-2026-06](docs/specs/bugs/uat-2026-06.md)) — they exist to enforce the root fix.
